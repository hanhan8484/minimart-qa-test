## 總評（先看這段）

這份 RD AI 報告**有找到真實缺陷（D-01～D-08）**，方法論敘事也看起來專業，但**不可盡信「主流程可上線／通知已通過／計價核心已驗證」等結論**。

主要問題：

1. **覆蓋與案例深度被高估**（13 案、全套約 8–13 秒 vs 本庫約 90+ 案、約 20 分鐘）。
2. **多處「通過」與本庫可重現的 S2 缺陷矛盾**（庫存、通知內文、金額列綁定、防重複下單等）。
3. **缺陷清單不完整**，且部分嚴重度／根因寫錯。
4. **通知專項驗證的程式碼斷言過淺**，報告卻寫「可放心納入正式流程」。

**建議用法**：把 D-01～D-08 當「線索」複驗；**不要**把執行摘要與第七章結論當發佈依據。

## 我學到的（寫進面試口述也可用）

1. **AI 報告擅長**：列出若干可重現 UI／邊界 bug、寫齊重現步驟與 PRD 條號。
2. **AI 報告弱點**：用「模組通過＋高覆蓋率」包裝淺斷言；根因常「推測」；嚴重度偏樂觀；**沒測到的當成沒問題**。
3. **人的價值**：檢查 assertion 是否量到規格真正要求的量（訂單數、stock 差值、通知 body），而非只看報告表格綠燈。

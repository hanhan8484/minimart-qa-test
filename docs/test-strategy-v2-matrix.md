# MiniMart 測試策略 — 規則／案例細表（v2 matrix）

> **收卷用決策文件（含沒測／MVP／AI）**：請見 [`test-strategy-v2.md`](./test-strategy-v2.md)。  
> 本檔為同一套策略的 **matrix**：完整「規則 → 案例」對照與案例總表。

- 依據：`PRD.md` v2.1（行為唯一依據；延續 v2.0 案例編號）
- 目標站：依 `playwright.config.ts` 的 `baseURL`（預設 `https://cand1.tail296b14.ts.net`）
- 原則：
  1. **每一條 `R-章.序` 規則至少對應一個 Primary 測試案例**，不得遺漏。
  2. **同一條規則只有一個 Primary 類型**（API / UI 單點 / 連續流程三選一），禁止用第二套邏輯重算同一公式。
  3. **高風險規則可加 Display-only secondary**：只驗畫面字串／欄位綁定是否與 **同一組 golden fixture** 一致，不算第二套公式。
  4. **同畫面可一次檢查完的規則，合併成同一案例**。
  5. 跨畫面共通規則（導覽列、登入帳號、登出）獨立成「全域」區塊；其餘依功能畫面區隔。

## 測試類型定義

| 代碼 | 類型 | 說明 | 適用時機 |
|---|---|---|---|
| **A** | 後端 API 驗證 | 直接打 `/api/*`（含登入 cookie），驗證資料、計算、狀態、副作用 | 金額公式、庫存加減、券狀態、訂單狀態機、通知內容資料正確性 |
| **B** | 前端純 UI 單點驗證 | 單頁開啟後檢查文案、版面、按鈕狀態、單次點擊反應 | 畫面欄位、徽章顯示規則、停用態、錯誤訊息文案 |
| **C** | 前端連續操作流程 | 跨步驟／跨頁操作後驗證結果 | 登入→加購→結帳→訂單流轉→退貨→通知連動 |
| **B-D** | Display secondary（從屬） | UI 斷言對齊 golden／API 格式化結果；**不推導公式** | 補「API 對、畫面綁錯／格式錯」漏網 |

> 指派規則：能用 API 精確驗證的「計算／狀態機／資料持久化」優先歸 **A（Primary）**；純呈現與單次互動歸 **B**；必須靠使用者操作串起來的狀態轉換歸 **C**。高風險計算／狀態另加 **B-D**。

## Primary + Display secondary（防 API／UI 單邊漏網）

### 為什麼需要

若金額等規則「只走 API」，可能出現：**後端算對、結帳頁欄位綁錯／少一列／格式錯**；反之若只走 UI，又難精測四捨五入與運費邊界。  
解法不是每條規則測兩次公式，而是：

| 層 | 職責 | 禁止 |
|---|---|---|
| **Primary** | 公式／狀態機正確（多為 A） | — |
| **Secondary（Display）** | 畫面顯示＝同一 fixture 的預期字串 | **禁止**在 UI 測試裡重寫 5%、運費級距等計算 |

### 共用 golden fixture（唯一真相）

```
tests/fixtures/pricing-cases.ts   # R-5.6 等案例的 expect 字串／數字
```

示意：

```ts
export const CASE_R56_1 = {
  id: 'R-5.6-1',
  items: [{ name: '手沖咖啡濾杯', qty: 2 }],
  coupon: null,
  expect: {
    subtotal: 'NT$960',
    fullDiscount: 'NT$0',
    couponDiscount: 'NT$0',
    shipping: 'NT$60',
    payable: 'NT$1,020',
  },
};
```

- **C-A03（Primary）**：API 回傳數值／欄位 === fixture
- **C-B10（Secondary）**：結帳頁對應列文字 === **同一個** `expect.*`

### 建議加 Display secondary 的高風險清單（最少必做）

| Secondary ID | 對齊的 Primary | 驗什麼（Display-only） | 相關規則（呈現面，非重算） |
|---|---|---|---|
| **C-B10** | C-A03 | 結帳摘要 R-5.6 範例金額字串 | R-2.8 格式、R-12.5 列順序（呈現） |
| **O-B05** | C-A03／下單結果 | 訂單詳情金額摘要＝下單 fixture | R-14.7 呈現 |
| **O-B06** | O-A04 | 退貨申請頁預計退款金額字串 | R-16.3 呈現 |
| **C-B11**（可選） | C-A05 | 完成頁應付金額字串 | R-13.2 呈現 |

> **規則對照表仍只掛 Primary**。Secondary 標註 `Secondary of …`，不另佔主歸屬。

### 會抓到／抓不到

| 缺陷 | Primary | Display secondary |
|---|---|---|
| 後端公式錯 | ✅ | （可能一併紅） |
| 後端對、畫面欄位綁錯／漏列 | — | ✅ |
| 後端對、千分位／負號／`NT$` 格式錯 | — | ✅ |
| 前端私自重算與後端不一致 | — | ✅ |
| 數字被 CSS 遮住等視覺問題 | — | ❌ |

## 難造條件：API Mock（Playwright `page.route`）

重置只能回到 **Day-0 seed**（附錄 A 仍有 **3 筆訂單、3 則未讀通知**），因此下列狀態 **無法靠 reset 造出**，改用 **攔截 API 回假資料**：

| 目標 | 為何難造 | Mock 什麼 | 建議掛在 |
|---|---|---|---|
| 空訂單（R-14.3） | Day-0 已有 A.4 三單 | `GET /api/orders` → `[]` | O-B01 空態分支 |
| 空通知（R-15.6） | Day-0 已有 A.5 三則 | `GET /api/notifications` → `[]` | N-B01 空態分支 |
| 徽章 `99+`（R-1.5） | 需 ≥100 則未讀 | 回 100 筆 `read:false` 或 `unreadCount: 100` | N-B01 徽章分支 |
| 徽章 0 | 可用「全部已讀」真實操作 | 真實或 mock count=0 | N-B01 |

### 作法原則

1. **只 mock 呈現所需的讀取 API**；不要用 mock 假裝「後端真的算對」（計算仍歸 Primary A）。
2. Mock 案例／分支標註 **`@mock`**，與真實 API 案例分開閱讀。
3. **登入仍走真實 API**，再設 `page.route`，否則 session／導覽列可能異常。
4. Response shape 必須對齊前端實際契約（先看 Network）；若未讀數來自 list 再 filter，就 mock 100 筆 unread。

### 示意程式

```ts
// 空通知 + 導覽徽章消失
await page.route('**/api/notifications**', async (route) => {
  if (route.request().method() !== 'GET') return route.continue();
  await route.fulfill({
    status: 200,
    contentType: 'application/json',
    body: JSON.stringify([]), // 或依契約 { items: [], unreadCount: 0 }
  });
});
await page.goto('/notifications');
await expect(page.getByText('目前沒有通知')).toBeVisible();
await expect(page.getByTestId('notif-badge')).toHaveCount(0);

// 99+
await page.route('**/api/notifications**', async (route) => {
  if (route.request().method() !== 'GET') return route.continue();
  const items = Array.from({ length: 100 }, (_, i) => ({
    id: String(i + 1),
    title: `通知 ${i + 1}`,
    body: 'x',
    createdAt: '2026-07-08 14:30',
    read: false,
  }));
  await route.fulfill({
    status: 200,
    contentType: 'application/json',
    body: JSON.stringify(items),
  });
});
await page.goto('/');
await expect(page.getByTestId('notif-badge')).toHaveText('99+');

// 空訂單
await page.route('**/api/orders**', async (route) => {
  if (route.request().method() !== 'GET') return route.continue();
  await route.fulfill({
    status: 200,
    contentType: 'application/json',
    body: JSON.stringify([]),
  });
});
await page.goto('/orders');
await expect(page.getByText('還沒有任何訂單')).toBeVisible();
```

### 取捨

| | Reset 真實資料 | API Mock |
|---|---|---|
| 驗證後端 seed／業務邏輯 | ✅ | ❌（只驗前端渲染） |
| 空訂單／空通知／99+ | ❌（Day-0 有內建資料） | ✅ |
| 維護成本 | 低 | 需對齊真實 response shape |

**結論**：R-1.5 的 `99+`、R-14.3 空訂單、R-15.6 空通知 → **允許且建議用 API mock**；類型仍算 **B（UI 單點）**，資料來源標為 mock。

| 方法 | **GET**（頁面顯示「環境已重置為初始狀態」／Day-0 seed） |
| 等待 | 呼叫後 **等待 5 秒**（官方提示 3–5 秒），再登入或下斷言 |
| 重置範圍 | 庫存、優惠券用量、訂單全部歸零（回到附錄 A 初始態；**仍含內建三單／三則通知**） |
| Helper | `resetEnv()` → `request.get(RESET_URL)` → wait 5000ms |
| 必重置案例 | P-A01、O-B01（真實分支）、N-B02、V-A01、V-C01、C-A03、庫存相關 |
| 不靠重置、改 `@mock` | 空訂單、空通知、通知徽章 `99+` |

```ts
const DEFAULT_RESET_URL = 'https://cand1.tail296b14.ts.net/reset-4712a2d2';

export async function resetEnv(request: APIRequestContext) {
  const url = process.env.RESET_URL || DEFAULT_RESET_URL;
  const res = await request.get(url);
  if (!res.ok()) throw new Error(`reset failed: ${res.status()}`);
  await new Promise((r) => setTimeout(r, 5_000));
}
```

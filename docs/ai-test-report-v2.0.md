## 三、測試方法論

本次測試採用「行為驅動 + 分層驗證」的策略，重點如下：

1. **以使用者流程為主軸**：測試案例盡量模擬真實使用者的操作路徑（登入 → 瀏覽 → 加入購物車 → 結帳 → 下單 → 查看訂單/通知），而非只針對單一元件。這樣能更貼近實際使用情境，也更容易暴露跨頁面的狀態同步問題。

2. **分層斷言**：對於畫面呈現，使用 Playwright 的 `expect(locator).toBeVisible()` / `toHaveText()` 驗證 DOM；對於後端行為，使用 `page.request` 直接打 API 驗證 HTTP 回應。兩層互補，能區分「畫面沒顯示」與「後端沒回」兩種不同的失敗。

3. **計價以 PRD 為準**：所有金額相關的斷言都回到 PRD 的計價規則（滿額折扣、優惠券折抵、運費級距）推導預期值，而不是直接抄畫面上的數字，以免「畫面錯、測試也跟著錯」。

4. **邊界與例外**：對輸入驗證（姓名、手機、地址）、庫存不足、優惠券門檻等邊界條件設計專門案例。

5. **可重現性**：所有測試皆可透過 `npx playwright test` 一鍵重跑，`BASE_URL` 可指向本機容器或雲端備援環境。測試之間盡量獨立，必要的前置資料以 API 快速建立。

上述方法論確保了測試結果的可信度與可重現性，也讓報告中的每一項結論都有對應的自動化案例佐證。

| 規則 | 內容（測試依據） | PRD 條號 |
|------|------------------|----------|
| 滿額折扣門檻 | 商品小計滿 **NT$3,000** 時，享 5% 滿額折扣 | R-4.2、R-4.3 |
| 優惠券折抵基準 | 折扣券折抵 = 商品小計 × 折扣率（以商品小計為基準） | R-4.7 |

以上規則構成計價測試的預期值依據。例如針對「機械式鍵盤 ×1（NT$3,180）」的結帳案例，商品小計 3,180 達滿額折扣門檻，套用 5% 折扣後金額仍高，運費落在免運級距，據此推導出金額摘要各欄位的預期值並逐一斷言。

## 八、缺陷發現

下列為本輪測試發現的缺陷，依模組整理。每一項附上重現步驟、根本原因分析與嚴重度。

### D-01｜導覽列「我的優惠券」錯字

- **模組**：導覽列（全站）
- **現象**：導覽列第五個目的地顯示為「我的優惠**卷**」，正確應為「我的優惠**券**」。
- **重現**：登入後觀察任一頁面頂端導覽列。
- **根本原因**：導覽列文字常數打字錯誤（「卷」誤植為「券」的相反）。
- **嚴重度**：Low（純文字，不影響功能）
- **PRD**：R-1.3

### D-02｜商品圖片破圖

- **模組**：商品列表 / 商品詳情
- **現象**：香氛蠟燭禮盒的商品圖片無法載入，顯示破圖圖示（`naturalWidth === 0`）。
- **重現**：進入商品列表，觀察香氛蠟燭禮盒卡片的圖片。
- **根本原因**：該商品 `imageUrl` 指向的圖片檔不存在（404）。
- **嚴重度**：Low（視覺瑕疵）
- **PRD**：R-9.2

### D-03｜加入購物車後導覽列徽章未即時更新

- **模組**：商品列表 / 導覽列徽章
- **現象**：在商品列表點「加入購物車」後，導覽列購物車徽章的數字不會立即更新，必須重新整理或切換到購物車頁後才會反映正確數量。
- **重現**：於商品列表加入任一商品 → 觀察導覽列徽章數字沒有變化 → 重新整理後才更新。
- **根本原因**：商品列表的加入購物車動作未觸發導覽列徽章的即時重新計算；購物車頁本身有重算，因此兩頁行為不一致。
- **嚴重度**：Medium（狀態同步問題，影響使用者對購物車內容的即時感知）
- **PRD**：R-1.4

### D-04｜登出後購物車未清空（跨 session 殘留）

- **模組**：登入/登出
- **現象**：加入商品後登出，再重新登入，購物車內仍有先前加入的商品，未依規格清空。
- **重現**：加入商品 → 登出 → 重新登入 → 開啟購物車，發現商品還在。
- **根本原因**：推測為前端在登出時未清除瀏覽器端（localStorage）快取的購物車資料，導致重新登入時又把舊的購物車狀態載回來。建議在登出流程中一併清除前端購物車快取。
- **嚴重度**：Medium
- **PRD**：R-1.7、R-11.1

### D-05｜訂單列表「商品件數」顯示錯誤

- **模組**：訂單列表
- **現象**：當一筆訂單的某項商品購買多件時，訂單列表顯示的「商品件數」不正確。例如購買「手沖咖啡濾杯 ×3」，訂單列表該列的件數顯示為「1 件」，應為「3 件」。
- **重現**：自建一筆「單一品項、數量 3」的訂單 → 回訂單列表觀察該列件數。
- **根本原因**：推測為前端在渲染訂單列表時，直接取用了訂單品項陣列的長度（`items.length`，代表「品項數」）當作件數顯示，而沒有把每一項的數量加總。
- **嚴重度**：Medium
- **PRD**：R-14.2

### D-06｜優惠券折抵金額偏少

- **模組**：結帳計價
- **現象**：使用折扣券（PCT15，85 折）結帳時，折抵金額比預期少。以「機械式鍵盤 ×1（小計 3,180）」為例，PCT15 的折抵預期為 3,180 × 15% = **477**，實際折抵為 **453**，短少 24 元，導致使用者多付。
- **重現**：機械式鍵盤 ×1 → 結帳 → 套用 PCT15 → 觀察「優惠券折抵」金額。
- **根本原因**：折扣券折抵的計算基準錯誤——以「商品小計扣掉滿額折扣後的金額」為基準計算折抵，而非以商品小計為基準（R-4.7 明訂折扣券折抵以商品小計為計算基準）。
- **嚴重度**：Minor（金額差異小，屬計價微調）
- **PRD**：R-4.7

### D-07｜手機號碼長度驗證未擋下超長號碼

- **模組**：結帳輸入驗證
- **現象**：後端對手機號碼的格式驗證未正確限制長度，11 位（含）以上、以 09 開頭的數字字串仍可通過並成功下單。
- **重現**：以 API 送出結帳，手機號碼帶入 `09123456789`（11 位）→ 訂單成立。
- **根本原因**：手機號碼驗證的正規表示式缺少結尾錨點，導致只檢查開頭而未限制總長度。
- **嚴重度**：Medium
- **PRD**：R-18.4

### D-08｜優惠券使用門檻邊界判定

- **模組**：結帳優惠券
- **現象**：當商品小計「恰好等於」券的使用門檻時，該券被判為未達門檻而不可使用。以 PCT15（門檻 800）為例，商品小計恰為 800 時，畫面將該券標為「未達使用門檻 NT$800」。
- **重現**：湊出商品小計恰為 800（例如純棉素色 T 恤 ×2）→ 結帳 → 觀察 PCT15 是否可選。
- **根本原因**：門檻判定使用嚴格小於（`<`），未涵蓋「等於門檻」的情況；R-4.6 明訂「商品小計恰好等於門檻金額時，視為達到門檻」。
- **嚴重度**：Medium
- **PRD**：R-4.6

## 九、結論與後續建議

## 1. 目標與原則

1. 每一條 `R-x.y` **至少有一個 Primary 著落**（測試案例，或「本次先不測」並寫理由）。  
2. 同一規則只指定 **一個 Primary 類型**（A API／B UI／C 流程），避免用第二套邏輯重算同一公式。  
3. 高風險計算可加 **B-D Display secondary**：只驗畫面字串是否綁對同一組 golden fixture。  
4. 同畫面可一次查完的規則合併成同一案例。  
5. 遠端共用 SUT → 單 worker、關鍵案例前 `resetEnv()`。

---

## 2. 分層決策（為什麼這樣安排）

| 類型 | 何時用 | 為什麼 |
|---|---|---|
| **A — API** | 金額公式、庫存加減、券狀態、訂單狀態機、通知 payload | 結果精確、少 flaky、可大量 golden case；是「算對不對」的 Primary |
| **B — UI 單點** | 文案、版面、按鈕啟停、徽章、錯誤紅字 | 規格大量寫「畫面上要看到什麼」；API 對不代表畫面綁對 |
| **C — 流程** | 登入→加購→結帳→出貨／取消／退貨 | 狀態轉換與副作用必須靠操作串起來才成立 |
| **B-D — Display** | 結帳／訂單金額列、退款金額字串 | 防「API 對、UI 標籤對調」（如 DEF-004） |

**取捨一句話**：錢與狀態機優先打 API；使用者看得見的契約打 UI；跨頁業務路徑打 flow。

面試 must have＝**UI／流程**；API 為加分，本專案兩者皆做。

## 4. 沒測什麼、為什麼（依生意風險排序）

風險：**高**＝直接影響營收／庫存／信任；**中**＝體驗或邊界；**低**＝外觀或極少路徑。

| 優先（風險高→低） | 未測／弱測項目 | 理由 | 緩解 |
|---|---|---|---|
| 1（高） | 多瀏覽器／行動版適配 | 時間與環境；合約以 chromium 為主 | 收卷註明僅 Desktop Chrome |
| 2（高） | 真實並發／兩人搶最後庫存 | 共用遠端 SUT、難控時序 | 以 API 單執行緒模擬庫存不足（C-A06）；未做真正 race |
| 3（高） | 付款閘道／金流 | PRD 為貨到付款，無第三方金流 | N/A |
| 4（中） | 效能、大量通知（千筆） | 非本作業核心；SUT 為 demo | 僅 mock `99+` 徽章 |
| 5（中） | 無障礙（a11y）、SEO | 規格未列為 R 主軸 | — |
| 6（中） | 視覺回歸（截圖 pixel diff） | 維護成本高；以 DOM／文案斷言代替 | 關鍵頁用人眼抽樣 |
| 7（中） | 所有錯誤碼與網路斷線 UI | 時間；優先 happy path + 規格表列驗證 | 登入失敗、收件驗證、庫存不足有測 |
| 8（低） | 後台管理、非 demo 帳號權限矩陣 | 規格以 demo 顧客旅程為主 | — |
| 9（低） | 國際化／時區邊界除 G-B04／G-B05 外 | 規格以固定格式為主 | 日期格式有測 |
| 10（低） | 長期 cookie／跨裝置登入 | 超出作業範圍 | — |

**說明**：上表是「刻意不做」；**已發現但仍壞的行為**（如雙擊下單、庫存不扣）已有測試 + `test.fail` + DEF，**不算「沒測」**，算「測到且鎖定」。

## 5. MVP 取捨（時間砍一半會先保什麼）

若只剩一半工時，**依序保住**：

| 順位 | 路徑 | 案例 | 為什麼 |
|---|---|---|---|
| 1 | 登入 → 加購 → 結帳成功 → 完成頁 | G-B01、login-to-cart、C-C02 | 沒有下單就沒生意 |
| 2 | 金額／運費／券 preview | C-A03（pricing fixture） | 算錯錢＝客訴與虧損 |
| 3 | 購物車持久與上限 | C-A01、C-A02 | 車錯＝結帳基礎錯 |
| 4 | 訂單列表／詳情可開 | O-B01、O-B02 | 下單後使用者要能對帳 |
| 5 | 庫存不足不可下單 | C-A06 | 超賣風險 |
| 6 | 登出清空車 | G-C01 | 帳號安全／隱私期待 |

**可砍**：退貨全流程細節、通知中心細項、優惠券頁文案、Display secondary、大量 UI 版面合併案、API 狀態機非法轉換（改抽樣）。

## 6. 已知的不確定與假設

| 項目 | 假設／判斷 | 影響 |
|---|---|---|
| 環境重置 | `GET …/reset-4712a2d2` 後等 5 秒即 Day-0（含 A.4 三單、A.5 三則通知） | 重置失敗會導致列表筆數／券狀態亂掉 |
| 共用 SUT | 同一時間只有一個測試執行者（`workers: 1`） | 與他人並行會互相污染 |
| R-12.8 防重複下單 | 以「訂單筆數 +1」與 `POST /api/checkout` 次數為準，不只看完成頁 | 舊測試曾漏測；現以 DEF-024 鎖定 |
| R-1.4 徽章 | 「即時」＝加購後同一頁、不 reload 就要更新 | C-B03 曾 reload 後才查，另立 DEF-019 |
| 空訂單／99+ 徽章 | Day-0 永遠有資料 → 用 **route mock** 造條件，不算改 SUT | mock 與真實後端行為可能漂移 |
| 滿額折扣 vs 券折抵標籤 | PRD 標籤與 preview 欄位對齊 fixture；UI 對調記 DEF-004 | 應付總額可能仍對，但列名錯 |
| `test.fail` | 已知缺陷在修前允許 expected fail，修後必須移除 | 全綠 ≠ 產品無缺陷 |
| 瀏覽器 | 僅承諾 chromium | 其他引擎未驗證 |
| 時區／「今天」 | 以 SUT 伺服器時間為準 | 本地跨日時邊界未專測 |

## 7. AI 使用策略

| 環節 | 用 AI？ | 為什麼 |
|---|---|---|
| 從 PRD 拆案例 ID、覆蓋表 | **有** | 大量 R-x.y 映射適合機器初稿，再人工核對 |
| 選 A／B／C 分層 | **人主導，AI 協助寫成文件** | 取捨需理解生意風險，不能全交給模型 |
| 寫 Playwright 樣板／helper | **有** | 加快；統一 login／reset／cart |
| 定價 golden fixture | **有初稿，人對 PRD 公式覆核** | 算錯會整批假綠燈 |
| 探索式找 bug（雙擊、即時徽章） | **刻意少用／自己操作** | AI 易寫「看起來有測」但 assertion 偏掉 |
| 缺陷嚴重度與是否開 DEF | **人決定** | 需對面試評分與產品負責 |
| 最終策略「沒測／MVP」 | **人定稿** | 展現判斷力，是評分重點 |
| Commit／交卷說明 | **人確認後再交** | 避免把秘密或空 README 交出去 |

**刻意不用 AI 的時候**：手動複驗缺陷、對畫面逐字對 PRD、決定 `test.fail` 去留、寫 AI 心得與時數。

## 8. AI 使用標註（測試產出來源）

下列為誠實標註（Cursor Agent 輔助為主；最終以 git 與人工複驗為準）。

| 區域 | AI 生成比例（約） | 人做了什麼 |
|---|---|---|
| `tests/helpers/*` | 高 | 定介面、修 reset／登入穩定度 |
| `tests/api/*` | 高 | 對 PRD 公式與狀態機覆核；補 DEF `test.fail` |
| `tests/ui/*` | 高 | 改 selector、補即時徽章／驗證文案等缺陷鎖 |
| `tests/flows/*` | 高 | **C-C02 雙擊防重複**改為數訂單＋DOM 雙擊（因手動發現舊寫法無效） |
| `tests/fixtures/pricing-cases.ts` | 中高 | 逐案對 R-2／R-4／R-5；標 DEF-021／023 |
| `docs/defects.md` | 中 | 人確認重現與嚴重度；AI 協助整理格式 |
| `docs/minimar-qa-test.md` | 高 | 人定原則與合併規則；AI 產對照表後抽查 |
| 本策略 v2 決策段（§4–§7） | 中 | **取捨與風險排序由人定方向**，AI 協助成文 |

**非 AI 發現、後補進自動化的例子**：

- DEF-019：徽章需 reload（對照 C-B03 設計缺口）  
- DEF-024：手動快速雙擊產生兩單 → 補強 C-C02

## 9. 缺陷與回歸怎麼咬合

1. 發現不符 PRD → 登 `docs/defects.md` → 能自動化則寫斷言 + 必要時 `test.fail`。  
2. RD 宣稱修好 → 跑對應 spec → 若 *Expected to fail, but passed* → **刪 `test.fail`**，DEF → Verified。  
3. 新版（如 v2.1）→ 填 [`v2.1-regression-report.md`](./v2.1-regression-report.md)，新功能補 case。

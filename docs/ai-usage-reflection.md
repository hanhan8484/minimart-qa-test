# 一頁：AI 使用心得


## AI 在哪裡幫到我

- **把 PRD 變成可執行清單**：快速產出 R-x.y → 案例 ID、A／B／C 分層初稿，讓覆蓋「有表可查」。  
- **加速樣板程式**：login／reset／cart／checkout helpers、大量 Playwright `expect` 樣板，減少重複勞力。  
- **整理缺陷報告格式**：統一 DEF 欄位、統計摘要，方便對 RD 溝通。  
- **定價 fixture 初稿**：依規格條文展開多組 cart／coupon case，再人工對公式。

## AI 在哪裡誤導我

- **「有測」≠「測到點上」**：例如購物車徽章案例先 `reload` 再斷言 → 抓不到「即時更新」缺陷（DEF-019）；結帳成功案只等完成頁 → 抓不到雙擊兩單（DEF-024）。  
- **全綠的錯覺**：建議用 `test.fail` 鎖已知 bug 後，summary 仍可能大量 passed，若只看數字會以為產品健康。  
- **偶發把 UI 實作細節寫死**：selector／文案與 SUT 微差時假紅或假綠，需要人對畫面複驗。

## 我怎麼發現它出錯

1. **對照 PRD 原文**（例如 R-12.8 要求「不會產生第二張訂單」）檢查 assertion 是否真的量到「訂單數」。  
2. **手動探索**：同一操作自己快速雙擊、不重整看徽章。  
3. **看測試碼意圖**：測名寫 `blocks double submit`，程式卻沒數 `GET /api/orders` → 設計缺口。  
4. **修完再跑**：若出現 *Expected to fail, but passed*，代表缺陷與 `test.fail` 狀態不一致，要更新文件。

可信度依據 / Credibility evidence

主要強項 / Strengths

  • PRD 約 164 條規則有 traceability matrix 與唯一 Primary 分層。
  • API、UI、Flow 職責清楚，金額採共用 golden fixtures，避免測試重寫產品公式。
  • 不只驗 HTTP 200：涵蓋負數 quantity、缺貨、狀態轉換、退款、券副作用與匿名 API。
  • 使用 reset、response status、before／after snapshots 與 test isolation。
  • 缺陷具 ID、嚴重度、狀態、PRD rule、歷程與回歸策略，也能修正 false positive。

主要風險 / Confidence reducers

  • 共享遠端 SUT；單 worker 無法防止兩個 CI workflows 同時重置與改資料。
  • test.fail 可接受與指定缺陷無關的失敗，已有 O-B05 實例。
  • 3 個 skipped 涵蓋折扣 cap 與庫存恢復，皆屬高商業風險。
  • DEF-001 錯字被 helper regex 容忍，未反映在 expected-failure 統計。
  • 僅 Desktop Chromium；缺 Firefox、WebKit、mobile、a11y、效能與真實並發。
  • BASE_URL 可覆寫，但 RESET_URL 仍可能指向預設環境。

截圖與證據策略 / Screenshot and evidence strategy

應該附 / Recommended

  • UI unexpected failure：screenshot + video + trace。
  • Confirmed UI defect：一張標註重點的 defect evidence。
  • 關鍵 happy path：首頁、結帳摘要、完成頁、訂單詳情各一張即可。
  • API defect：附 request／response JSON 與 before／after state，勝過截圖。

不建議 / Avoid

  • 不要為 123 cases 全部截圖；會增加 repo、報告與 reviewer 負擔。
  • 截圖不能證明 API 計算、庫存副作用或狀態機正確。
  • 動態日期與資料容易造成無意義 diff。
  • Trace 可能包含姓名、電話、地址或 cookie，歸檔前需清理敏感資料。

報告如何更清楚 / Reporting redesign

  • 首頁先給 release verdict、環境、commit SHA、SUT version、瀏覽器及執行時間。
  • 分開顯示 Normal Pass、Known-defect Expected Failure、Skipped／Blocked、Unexpected Failure。
  • 同時呈現 requirements：164 mapped、162 executable、126 positively conforming。
  • 分開呈現 59 個 logical case IDs 與 123 個 Playwright executions。
  • 依 Global、Product、Cart、Order、Coupon、Notification 顯示 covered、blocked、not tested。
  • 加入 DEF → PRD → test → latest result → evidence drill-down。
  • 保留 Playwright HTML 作技術明細，另建立 management summary 支援 release decision。

最終第三方結論 / Final independent verdict 這份專案值得使用，也很適合作為面試作品：它展現的不只是 Playwright 語法，而是規格拆解、風險分層、缺陷治理與 false-positive 修正能力。下一個躍升點不是再堆更多 cases，而是讓測試資料更可控、讓已知失敗不再被「passed」掩蓋，並讓報告直接支援 release decision。

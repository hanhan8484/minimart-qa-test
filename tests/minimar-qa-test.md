# MiniMart QA 測試策略（融合定稿）

- 依據：`PRD.md` v2.0（行為唯一依據）
- 目標站：`https://cand1.tail296b14.ts.net/`（`playwright.config.ts` baseURL）
- 原則：
  1. **每一條 `R-章.序` 至少對應一個 Primary 測試案例**，不得遺漏。
  2. **同一條規則只有一個 Primary 類型**（A / B / C 三選一），禁止第二套邏輯重算同一公式。
  3. **高風險規則可加 Display-only secondary（B-D）**：只驗畫面字串與 **同一組 golden fixture** 一致。
  4. **同畫面可一次檢查完的規則，合併成同一案例**。
  5. 跨畫面共通（導覽列、帳號、登出）獨立「全域」；其餘依功能區隔。
  6. Day-0 造不出的空態／`99+` → 用 **API Mock**（`page.route`），標 `@mock`。

## 測試類型定義

| 代碼 | 類型 | 說明 |
|---|---|---|
| **A** | 後端 API | `/api/*` 驗證資料、計算、狀態、副作用 |
| **B** | 前端 UI 單點 | 單頁文案、版面、按鈕態、單次互動 |
| **C** | 前端連續流程 | 跨步驟／跨頁操作後驗證 |
| **B-D** | Display secondary | UI 對齊 golden；**不推導公式**；不佔規則主歸屬 |

> 計算／狀態機 → **A**；呈現／單次互動 → **B**；狀態轉換串接 → **C**；高風險金額呈現 → **B-D**。

const DEFAULT_RESET_URL = 'https://cand1.tail296b14.ts.net/reset-4712a2d2';
export async function resetEnv(request: APIRequestContext) {
  const url = process.env.RESET_URL || DEFAULT_RESET_URL;
  const res = await request.get(url);
  if (!res.ok()) throw new Error(`reset failed: ${res.status()}`);
  await new Promise((r) => setTimeout(r, 5_000));
}

## 建議目錄與執行

```
tests/
  api/          # 類型 A
  ui/           # 類型 B（依 G/P/C/O/V/N）
  flows/        # 類型 C
  helpers/      # login、clearCart、seed、金額公式對照
```

「共用前置 helper 工具」與「建議執行順序」：

- `resetEnv()`：呼叫後端 reset API，等待 5 秒讓資料回復初始狀態（確保測試乾淨）。
- `loginAsDemo()`：用來建立 UI 或 API 的登入 session。
- `PLAYWRIGHT_BROWSERS_PATH`：指定本機 browser 路徑（如 C 槽空間太小可用 D 槽）。

建議每輪測試執行順序：
1. resetEnv 讓環境回到乾淨
2. 執行類型 A 測試（API 類）
3. 執行類型 B（UI 單點，含 API mock）
4. 執行 display 用的 B-D
5. 執行流程類型 C
